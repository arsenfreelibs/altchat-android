# ТЗ: Авто-прокси при блокировке релея (Android)

## 1. Цель
Если клиент **не может достучаться до нашего релея** (IMAP/SMTP сервер аккаунта), но **интернет в целом работает**, приложение должно автоматически уйти через один из заранее вшитых HTTP-прокси. Когда прямая связь с релеем восстановится — вернуться к прямому соединению.

Это перенос уже реализованной фичи с iOS. Поведение должно совпадать.

## 2. Ключевые факты о ядре (общие для iOS и Android)
- Прокси хранится в конфиге аккаунта: `proxy_url` (список, строки разделяются `\n`) и `proxy_enabled` (`"0"/"1"`).
- **Ядро использует только ПЕРВУЮ строку `proxy_url`.** Остальные строки — просто список для UI. → Ротацию между прокси делаем сами: ставим нужный URL первой строкой и перезапускаем IO.
- Прокси — **HTTP с авторизацией**. Формат строки:
  ```
  http://USER:PASS@HOST:8000
  ```
- Состояние связи с релеем: `dcContext.getConnectivity()` → значения `DC_CONNECTIVITY_*` (1000…4000). «Подключено» = `>= DC_CONNECTIVITY_CONNECTED` (4000).
- Событие смены состояния: `DC_EVENT_CONNECTIVITY_CHANGED` (через `DcEventCenter`).
- Хост/порт релея для прямой проверки: `getConfig("configured_mail_server")` и `getConfigInt("configured_mail_port")`.
- Управление IO: `DcAccounts.startIo()` / `stopIo()` / `maybeNetwork()`. Эквивалент iOS-овского `restartIO()` — это перезапуск IO у `DcAccounts` (как делает экран настроек прокси при смене прокси).

## 3. Вшитые прокси и тайминги (константы)
```
// Реальные прокси НЕ хранятся в репозитории — задаются при сборке/рантайме
// (gitignored gradle property → BuildConfig, либо удалённый конфиг).
PROXIES = [
  "http://USER:PASS@HOST:8000",
  // ...
]
GRACE_SECONDS          = 20    // сколько релей должен быть «без связи» до включения прокси
PROXY_TRY_SECONDS      = 30    // время на один прокси, потом ротация
DIRECT_RECHECK_SECONDS = 240   // как часто на прокси перепроверяем прямую связь
BACKOFF_SECONDS        = 120   // пауза после неудачного перебора всех прокси
PROBE_TIMEOUT_SECONDS  = 8
INTERNET_PROBE_URL     = "https://www.google.com/generate_204"  // должен вернуть HTTP 204/любой ответ
```

## 4. Стейт-машина
Три состояния, все переходы — на одном фоновом потоке (Handler/Executor), без UI-потока.

**`DIRECT`** (прокси выключен, наблюдаем за релеем):
- На событие connectivity: если `getConnectivity() >= CONNECTED` — отменить ожидание.
- Если связь упала (`< CONNECTED`) и таймер ещё не запущен — запланировать проверку через `GRACE_SECONDS`.
- По истечении grace: если связь всё ещё `< CONNECTED` → **проба интернета** (`INTERNET_PROBE_URL`):
  - интернет есть → перейти в `ENGAGING`, выбрать **случайный** стартовый прокси;
  - интернета нет → это общий обрыв сети, прокси не трогаем, перепланировать проверку ещё раз через `GRACE_SECONDS`.

**`ENGAGING`** (включили прокси-кандидат, ждём подключения):
- `engageCurrentProxy()`: домержить вшитые прокси в `proxy_url` (сохранив пользовательские, без дублей), поставить выбранный URL **первой строкой**, `proxy_enabled = 1`, перезапустить IO. Выставить флаг «прокси включили мы» (см. §6).
- На событие connectivity `>= CONNECTED` → перейти в `ENGAGED`.
- Таймер `PROXY_TRY_SECONDS`: если ещё не подключились → ротация на следующий прокси (`(idx+1) % N`), снова `engageCurrentProxy()`.
- Если перебрали все N без успеха → пауза `BACKOFF_SECONDS`, затем новый заход (случайный прокси).

**`ENGAGED`** (прокси работает, ждём восстановления прямой связи):
- На событие connectivity `< CONNECTED` (прокси отвалился) → начать перебор заново (`ENGAGING`).
- Таймер `DIRECT_RECHECK_SECONDS`: **прямая** TCP-проба к `configured_mail_server:configured_mail_port` (см. §5), минуя прокси:
  - доступен → `proxy_enabled = 0`, перезапуск IO, снять флаг → `DIRECT`;
  - недоступен → перепланировать проверку.

## 5. Сетевые пробы (Android-специфика)
- **Проба интернета:** `GET https://www.google.com/generate_204` через `HttpURLConnection`/OkHttp с таймаутом `PROBE_TIMEOUT_SECONDS`. Успех = получен любой HTTP-ответ без исключения. (Опционально можно дополнительно сверяться с `ConnectivityManager` + `NetworkCapabilities.NET_CAPABILITY_VALIDATED`.)
- **Прямая проба релея:** `new Socket()` + `socket.connect(new InetSocketAddress(host, port), timeoutMs)`. Обычный `Socket` не ходит через HTTP-прокси ядра, поэтому проверяет именно прямую доступность. Закрыть сокет в `finally`.

## 6. Гард (важно — чтобы не воевать с пользователем)
- Храним флаг `autoProxyEngaged` (SharedPreferences) — «прокси включили мы автоматически».
- Если `proxy_enabled == 1`, но `autoProxyEngaged == false` → пользователь сам включил прокси: **ничего не трогаем**, остаёмся в `DIRECT` без вмешательства.
- Авто-выключать прокси (`disengage`) можно только если `autoProxyEngaged == true`.
- Действуем только на **выбранном сконфигурированном** аккаунте (`isConfigured()`).

## 7. Точки интеграции
- Создать класс `AutoProxyManager` (аналог iOS-овского), внедрить в Application/основной сервис после `startIo()`.
- Подписать его на `DC_EVENT_CONNECTIVITY_CHANGED` через `DcEventCenter`.
- При возврате приложения на передний план / смене сети — дёргать `reevaluate()` (необязательно, событие connectivity и так придёт).
- При старте: если `autoProxyEngaged == true` (включали в прошлой сессии) — стартовать сразу в `ENGAGED` и запустить `DIRECT_RECHECK`.

## 8. Краевые случаи
- Нет интернета вообще → прокси не включаем (бессмысленно, экономим прокси).
- Прокси-кандидат не подключился за 30 c → следующий; полный круг без успеха → бэкофф 2 мин.
- Креды вшиты в открытом виде (как и на iOS — требование «вшить»). При желании позже вынести в обфускацию/удалённый конфиг.
- Мультиаккаунт — пока только выбранный аккаунт.

## 9. Критерии приёмки (как тестировать)
1. На сконфигурированном аккаунте заблокировать хост релея при живом интернете → через ~20 c прокси включается сам, в списке прокси появляются 4 вшитых, тумблер встаёт в ON, connectivity доходит до Connected.
2. Снять блокировку релея → в течение ~3–5 мин прямая проба проходит, прокси сам выключается, работа идёт напрямую.
3. Полностью отключить сеть → прокси НЕ включается.
4. Пользователь вручную включил свой прокси → менеджер его не выключает и не подменяет.
5. На разных устройствах стартовый прокси разный (рандом), нагрузка размазана по 4 серверам.
